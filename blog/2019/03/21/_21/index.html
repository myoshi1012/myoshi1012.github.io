# aws-cdkとは

https://docs.aws.amazon.com/CDK/latest/userguide/what-is.html
ここに書いてある

要はプログラミング言語(js,ts,java,.net)でCloudformationのスタックを作成するためのフレームワーク。 長くなりがちなyamlテンプレートを書かずにCloudformationスタックが良い感じに生成できるらしい。

# コンセプト

ドキュメントに書いてある通りなんだけど、理解が浅いのでまとめておく。

# Constructs

  • cloud componentと考えることができる
  • ConstructsはContructsを含めることができる
  • AWS CDKの中のものは全部Constructs
  • SDKやCloudformationで提供されているようなAWSのリソースはConstructs Libraryとして提供される

みたいなことが書いてあるので、Contructsを組み合わせてアプリケーションを作っていくことになるんだと思う。
レゴみたいな感じか（適当）

# Apps と Stacks

# Apps

stackの集合体。完成品みいたいな感じか。

# Stacks

AWSの環境にデプロイできるConstructsとある。
ConstructsがClassとして定義されるので、つまるところインスタンスって感じか。

お仕事ではbeanstalkをよく使うので、とりあえずbeanstalkのapplicationとenvironmentを作ってみる。
(尚、昨日マージされたexampleが存在するのでこれを コピペ 参考にします🐢…)

lib/my_first_cdk-stack.ts

import cdk = require('@aws-cdk/cdk');
import elasticbeanstalk = require('@aws-cdk/aws-elasticbeanstalk');

export class MyFirstCdkStack extends cdk.Stack {
  constructor(scope: cdk.Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    //objects for access parameters
    const node = this.node;

    const solutionStackName = node.getContext("solutionStackName")


    const app = new elasticbeanstalk.CfnApplication(this, 'Application', {
      applicationName: "MyApp"
    });

    new elasticbeanstalk.CfnEnvironment(this, 'Environment', {
      environmentName: 'MySampleEnvironment',
      applicationName: app.applicationName,
      solutionStackName: solutionStackName
    });
  }
}

context（コマンドラインのパラメータ）でplatformを指定するようにしました。(実運用ではplatformはアプリケーションごとに決まってると思うので、わざわざパラメータにする必要はなさそう。)
存在するsolution stack名はaws elasticbeanstalk list-available-solution-stacks --query 'SolutionStacks[]'で一覧取得できる。 ビルドして作成されるCloudformationスタックのtemplateを確認してみる。

# 雑感

CloudformatioinのYAMLのtemplateをガリガリ書くよりは楽そう。ほぼプログラミングなので、アプリケーションエンジニアもAWSのリソース管理をしやすいと思う。ただyamlと違ってぱっと見何のリソースが存在するのかわかりにくいので、typescriptのコメントを活用してドキュメント作るなり何かしら対処しないと管理が煩雑になりそう。
beanstalkは仕組み上、環境差分が作りにくかったりするけどcdkで解消できそうだし、マルチアカウント、マルチリージョンも実現しやすそうなので次回はその辺りを検証してみたい。
